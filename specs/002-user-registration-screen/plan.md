# Implementation Plan: User Registration Screen

## Summary

Implement a secure, WCAG 2.1 AA-compliant user registration form for the patient portal. The feature enables new patients to create accounts via name, email, and password inputs with real-time client-side validation, error handling without sensitive exposure, and backend persistence through Next.js API routes. Designs reference Google Stitch prototypes for web and mobile; MVP focuses on web with TDD approach enforced per Constitution Principle II.

## Technical Context

**Language/Version**: TypeScript 5.x with Next.js 14.2+  
**Primary Dependencies**: React 18+, React Hook Form, Zod, Tailwind CSS  
**Storage**: User credentials (backend PostgreSQL via API; client-side: none except form state)  
**Testing**: Jest + React Testing Library (unit/integration), Playwright (E2E), coverage ≥80%  
**Target Platform**: Web (desktop + mobile responsive); mobile app as Phase 2 expansion  
**Project Type**: Next.js web application (frontend feature) with API route integration  
**Performance Goals**: Page load ≤2.5s LCP, form interaction ≤100ms, submit response ≤200ms p95  
**Constraints**: <50KB bundle impact, ≥55fps mobile interactions, no sensitive data in logs  
**Scale/Scope**: 1 screen, ~5 form fields, 2 async flows (validation + submission), ~400 LOC feature code

## Phase 0: Research & Unknowns

### Known Unknowns from Spec Section 8

1. **Form Library Choice**
   - Decision needed: React Hook Form vs. Formik vs. simple useState?
   - Impact: Component complexity, bundle size, testing strategy
   - Research task: Compare bundle impact + feature completeness for this simple 3-field form
   - Recommendation (pending research): React Hook Form (lightweight, ≤10KB, integrates well with Zod)

2. **Validation Library**
   - Decision needed: Zod vs. Yup vs. native validation?
   - Impact: Type safety, schema reusability (backend + frontend), bundle size
   - Research task: Verify Zod integration with React Hook Form; check backend compatibility
   - Recommendation (pending research): Zod (TypeScript-first, already common in React ecosystem)

3. **Route Naming Convention**
   - Decision needed: `/register` vs. `/auth/register` vs. `/signup`?
   - Impact: URL strategy, SEO, backend routing alignment
   - Constraint: Must align with backend routes + project routing convention (see 001-website-project-setup route-definitions.ts)
   - Research task: Check existing route conventions in route-definitions.ts
   - Recommendation (pending research): `/register` (simplest; auth group TBD in Phase 1)

4. **Success Redirect Destination**
   - Decision needed: After successful registration, redirect to login, onboarding, or dashboard?
   - Impact: UX flow, backend dependencies, session management
   - Constraint: Email verification deferred to Phase 2; assume no email confirmation for MVP
   - Research task: Confirm with backend / UX whether auto-login or redirect-to-signin for MVP
   - Recommendation (pending research): Redirect to `/login` with success message (does not require email verification)

5. **Backend API Contract Confirmation**
   - Decision needed: Exact error codes, response format, password requirements?
   - Impact: Error handling specificity, security requirements
   - Spec assumption: POST /api/auth/register, error codes EMAIL_EXISTS + INVALID_INPUT
   - Research task: Verify backend API specification (may need coordination with backend team)
   - Recommendation (pending research): Codify contract in contracts/registration-api.md; accept backend spec as-is

6. **Design Token Extraction from Google Stitch**
   - Decision needed: Map colors, spacing, fonts from Stitch prototypes to Tailwind tokens?
   - Constraint: Cannot directly access Stitch interactive prototype (design tool SPA)
   - Workaround: Extract token mapping from screenshot/design sync docs or confirm with designer
   - Research task: Request Stitch export or design handoff document from UX team
   - Recommendation (pending research): Use existing Tailwind defaults (primary, neutral, error tokens); confirm with designer if custom tokens needed

## Phase 1: Design & Contracts

### 1.1 Data Model & Schema

**Input**: spec.md Section 4 (Technical Context) + research.md (validated technologies)

**Outputs**: data-model.md, contracts/registration-api.md

#### Registration Data Model

**Entity**: RegistrationPayload

```typescript
// User input (client-side form)
interface RegistrationInput {
  fullName: string;         // 2-256 chars, alphanumeric + spaces
  email: string;            // RFC 5322 simplified; unique constraint
  password: string;         // 8+ chars, uppercase + lowercase + digit + special
  confirmPassword?: string; // Optional for explicit UX confirmation (TBD in design)
}

// Server response (success 201)
interface RegistrationSuccess {
  userId: string;
  email: string;
  message: string; // "Account created successfully"
}

// Server response (error 400/409/500)
interface RegistrationError {
  error: string;     // User-friendly message
  errorCode: string; // EMAIL_EXISTS | INVALID_INPUT | SERVER_ERROR | NETWORK_ERROR
  field?: string;    // Optional: which field failed (email, password, etc.)
}
```

**State Machine**: Form submission flow

```
[Idle] 
  ↓ (user fills form)
[Validating] → [Valid] or [Invalid]
  ↓ (user submits valid form)
[Submitting]
  ↓ (success 201)
[Success] → (redirect after delay)
  ↓ (error 4xx/5xx)
[Error] → (show error message, allow retry)
```

#### Validation Schema (Zod)

```typescript
const registrationSchema = z.object({
  fullName: z
    .string()
    .min(2, "Name must be at least 2 characters")
    .max(256, "Name must not exceed 256 characters")
    .regex(/^[a-zA-Z\s'-]+$/, "Name can only contain letters, spaces, hyphens, and apostrophes"),
  email: z
    .string()
    .email("Invalid email address")
    .max(254, "Email must not exceed 254 characters"),
  password: z
    .string()
    .min(8, "Password must be at least 8 characters")
    .regex(/[A-Z]/, "Password must contain an uppercase letter")
    .regex(/[a-z]/, "Password must contain a lowercase letter")
    .regex(/[0-9]/, "Password must contain a number")
    .regex(/[^a-zA-Z0-9]/, "Password must contain a special character"),
});

type RegistrationInput = z.infer<typeof registrationSchema>;
```

### 1.2 API Contract

**Output**: contracts/registration-api.md (detailed specification for backend integration)

#### Endpoint: POST /api/auth/register

**Request**:
```http
POST /api/auth/register HTTP/1.1
Content-Type: application/json
Content-Length: 123

{
  "fullName": "Jane Doe",
  "email": "jane.doe@example.com",
  "password": "SecurePass123!"
}
```

**Success Response (201 Created)**:
```json
{
  "userId": "usr_abc123def456",
  "email": "jane.doe@example.com",
  "message": "Account created successfully"
}
```

**Error Responses**:

- **400 Bad Request** (validation error):
  ```json
  {
    "error": "Invalid email address",
    "errorCode": "INVALID_INPUT",
    "field": "email"
  }
  ```

- **409 Conflict** (email exists):
  ```json
  {
    "error": "Email already registered. Try login or use another email.",
    "errorCode": "EMAIL_EXISTS"
  }
  ```

- **500 Internal Server Error** (server fault):
  ```json
  {
    "error": "Unable to create account. Please try again later.",
    "errorCode": "SERVER_ERROR"
  }
  ```
